# Case Proof And Evidence Acceptance Checklist

## System Read

- [ ] Er is een expliciete scheiding tussen deliverable-proof, trustproof, validation evidence, case candidate, approved case proof en reference-ready.
- [ ] Sample-output, demo en validatie kunnen niet ongemerkt opschuiven naar marktbewijs.
- [ ] ExitScan staat expliciet als default publieke case-anchor en RetentieScan leest als geldige first-buy evidence-route bij een expliciete actieve behoudsvraag, zonder de verification-first grens te verliezen.
- [ ] De repo maakt duidelijk dat echte case-proof nog pas buyer-facing wordt na approval.

## Capture Read

- [ ] De learninglaag legt koopreden, eerste managementwaarde, eerste actie, reviewmoment en vervolguitkomst vast.
- [ ] Quote-potentie, referentiepotentie, outcome-kwaliteit, toestemming en supporting artifacts krijgen persistente capture.
- [ ] Closure-status maakt expliciet onderscheid tussen lesson only, internal proof only, sales usable, public usable en rejected.
- [ ] Geen buyer-facing caseclaim is mogelijk zonder provenance en context.

## Format And Approval Read

- [ ] Mini-case, anonieme case, quote card, reference note, outcome summary en objection-proof snippet hebben een vaste plek.
- [ ] De approvalflow `draft -> internal_review -> claim_check -> customer_permission -> approved` is zichtbaar in docs en data.
- [ ] Named quote, logo, ROI-claim of named case kan niet zonder expliciete toestemming.
- [ ] Anonieme proof blijft de default eerste publieke laag.

## Surface Parity

- [ ] Site, pricing en trust blijven sample-first totdat approved case-proof beschikbaar is.
- [ ] Sales gebruikt geen case-anekdotes buiten de approvallaag.
- [ ] Demo en showcase worden nergens verkocht als klantbewijs.
- [ ] Buyer-facing cards en docs maken zichtbaar welke evidence-tier een asset draagt.

## Governance Read

- [ ] Er is een source-of-truth planbestand voor deze tranche.
- [ ] Referentiedocs, ops-playbooks en tests noemen dezelfde evidence-taxonomie.
- [ ] `PROMPT_CHECKLIST.xlsx` is bijgewerkt op feitelijke repo-uitvoering.
- [ ] Toekomstige proofwijzigingen hebben een duidelijke parity- en acceptance-route.
