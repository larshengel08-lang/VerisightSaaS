# Loep Culture Assessment Listening Program Operations

**Date:** 2026-05-20  
**Status:** wave-1.5 design spec  
**Primary scope:** `Loep Culture Assessment / Loep Cultuurbeeld`  
**Canonical route id:** `culture_assessment`

---

## 1. Purpose

This spec defines the **Listening Program Operations** layer for `Loep Culture Assessment / Loep Cultuurbeeld`.

Its purpose is to make the annual baseline operationally executable as a bounded listening program without turning the route into a generic survey operations platform.

This spec must answer:

- how the route moves from launch preparation to closure and release
- what minimum operating cadence must exist before the first pilot
- which roles and checkpoints are required
- how reminders, closure, release, and board-read handoff are controlled
- what the first pilot may and may not do

---

## 2. Strategic Role

Inside the operating system, this stream exists because launch credibility depends not only on method and outputs, but also on whether the route can actually be run with discipline.

Without this layer, the product risks:

- feeling premium in deck form but improvised in real operation
- releasing results without a repeatable cadence
- overloading the first pilot with undefined delivery decisions

This stream therefore turns:

- annual baseline concept

into:

- annual baseline operating model

---

## 3. Operating Stance

The route is operated as:

- one bounded annual baseline program
- one guided launch-to-close cadence
- one controlled results release sequence
- one guided board-read handoff

The route is not operated as:

- a continuous survey utility
- a pulse engine
- a self-serve admin factory
- a multi-track lifecycle survey stack

---

## 4. Canonical Program Phases

The bounded V1 program contains eight phases.

1. scope and stakeholder alignment
2. campaign configuration and readiness check
3. respondent preparation and import
4. launch and invite wave
5. response monitoring and reminder cadence
6. closure and release check
7. board output preparation
8. board-read and governed follow-up handoff

---

## 5. Required Operating Roles

The route must assign at least these operational roles:

- executive sponsor
- HR implementation partner
- Loep delivery lead
- release owner
- board-read facilitator

The first pilot may combine roles if ownership remains explicit.

---

## 6. Pre-Launch Readiness

Pre-launch readiness must confirm:

- method source is locked
- output pack is demo-ready
- trust/procurement FAQ is ready
- role and release ownership are explicit
- thresholds and governance rules are understood
- first pilot constraints are accepted

No launch should begin if these conditions are vague or still disputed.

---

## 7. Respondent Preparation

The operations layer must define:

- what respondent data is required
- how import quality is checked
- which organizational fields are needed for safe segment reporting
- how incomplete or unsafe organizational metadata is handled

This phase must stay bounded:

- no dynamic survey design
- no custom module branching
- no uncontrolled local targeting logic

---

## 8. Invite and Launch Cadence

The route must define:

- launch message owner
- first invite timing
- reminder cadence
- closing notice timing

V1 minimum operating bar:

- launch invite exists
- reminder rhythm exists
- closure notice exists
- buyer and delivery owner know when the campaign is considered closed

---

## 9. Response Monitoring

Response monitoring must be operationally defined, even if the tooling remains light.

The route must be able to answer:

- who monitors coverage
- what counts as insufficient coverage risk
- when reminders are sent
- when segment ambition should be reduced for safety

This is not a live analytics product.
It is a guided operating discipline.

---

## 10. Closure and Release Protocol

Closure and release are separate steps.

### 10.1 Closure

Closure confirms:

- response collection has ended
- late submissions are bounded
- the baseline is ready for release assessment

### 10.2 Release

Release confirms:

- valid scoring basis exists
- minimum-n and release rules are satisfied
- unsafe layers remain suppressed
- release owner approves normal output

No normal board or HR output should render before release conditions pass.

---

## 11. Board Output Preparation

Before the board-read, the delivery layer must prepare:

- board report
- executive one-pager
- boardroom deck
- facilitator script
- governed HR layer where applicable

The route must define expected preparation time per client and what remains manual, semi-automated, or automated.

---

## 12. Board-Read Handoff

The operating model must define:

- who facilitates the board-read
- who attends
- what gets prepared in advance
- how follow-up is documented
- when Pulse remains out of scope versus follow-on eligible

The board-read is required to remain guided in the first pilot.

---

## 13. First Pilot Constraints

The first pilot must remain bounded by these rules:

- maximum one organization at a time
- no benchmark
- no manager layer unlock
- open text only if clustering is safe
- maximum two segment dimensions
- board-read always guided
- no custom questions

These are not optional preferences.
They are operating constraints.

---

## 14. Required Operating Assets

This spec must produce:

- launch readiness checklist
- stakeholder mapping template
- invite and reminder cadence template
- closure and release checklist
- board-read handoff checklist
- first pilot constraint note

## 15. Acceptance Criteria

This spec is complete only when:

1. the canonical phases are explicit
2. required operational roles are explicit
3. pre-launch readiness is explicit
4. invite, monitoring, and closure cadence are explicit
5. release protocol is explicit
6. board output preparation expectations are explicit
7. board-read handoff is explicit
8. first pilot constraints are explicit
9. the first pilot can be run without inventing the cadence ad hoc

---

## 16. Out of Scope

This spec does not:

- create a general survey operations platform
- introduce Pulse runtime
- define lifecycle survey programs
- create self-serve reminder orchestration
- create automated action management
