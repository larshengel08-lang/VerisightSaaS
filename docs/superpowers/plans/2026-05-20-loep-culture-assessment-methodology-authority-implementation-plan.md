# Loep Culture Assessment Methodology Authority Implementation Plan

**Date:** 2026-05-20  
**Status:** first-wave implementation plan  
**Primary scope:** `Loep Culture Assessment / Loep Cultuurbeeld`  
**Canonical route id:** `culture_assessment`

---

## 1. Purpose

This plan translates the `Methodology Authority System` spec into a bounded execution sequence.

Its purpose is to lock:

- method source of truth
- versioning and governance ownership
- claim-safe interpretation copy
- evidence and non-proof boundaries

without broadening the product into a psychometric program or benchmark project.

---

## 2. Execution Rules

1. one fixed method core remains authoritative
2. no silent method drift
3. no causal, predictive, benchmark, or manager-rating claims may appear
4. questionnaire, scoring, and interpretation copy must remain linked
5. every work package must produce a reviewable artifact

---

## 3. Work Package Sequence

### WP1 - Method Source Lock

Goal:

- define the canonical source of truth for the method

Deliverables:

- named method core reference
- source-of-truth file map
- ownership note for method source

Dependencies:

- approved methodology authority spec

Acceptance:

- one canonical method source is identified
- route id remains `culture_assessment`
- no duplicate method truth remains across docs without reference back

### WP2 - Domain and Index Lock

Goal:

- make domain logic and Culture Index calculation operationally unambiguous

Deliverables:

- canonical domain register
- index calculation note
- recurring theme pair register

Dependencies:

- WP1

Acceptance:

- ten domains are locked
- 8-of-10 valid-domain rule is explicit
- equal-weighted index rule is explicit
- recurring theme pairs are bounded and versioned

### WP3 - Response Validity and Release Boundary Lock

Goal:

- separate scoring eligibility from release eligibility

Deliverables:

- valid response policy note
- safe release rule note
- method-to-governance boundary summary

Dependencies:

- WP1
- WP2

Acceptance:

- valid response versus safe release is explicit
- no normal output is possible when either side fails
- release logic language is claim-safe and buyer-safe

### WP4 - Item-Intent and Questionnaire Writing Guardrails

Goal:

- prevent future overlap, drift, and shallow copy edits

Deliverables:

- item-intent slots register
- questionnaire writing rules
- reverse-scoring guardrails

Dependencies:

- WP2

Acceptance:

- all ten domains have four intent slots
- reverse scoring rules are explicit
- questionnaire changes are treated as method version events

### WP5 - Commercial Method Story and Method Note

Goal:

- make the method commercially explainable without overclaiming

Deliverables:

- canonical Dutch commercial method paragraph
- claim-safe method note
- short sales explanation

Dependencies:

- WP2
- WP3

Acceptance:

- commercial explanation stays aligned with method logic
- no unsupported scientific-certification claim appears
- governance is framed as quality, not friction

### WP6 - Versioning and Governance Log

Goal:

- make all future method changes auditable

Deliverables:

- method versioning template
- comparability statement template
- governance log owner note

Dependencies:

- WP1
- WP2
- WP3

Acceptance:

- no silent method drift remains acceptable
- version event types are explicit
- governance-log owner is explicit

### WP7 - Evidence Maturity and Review Gate

Goal:

- define how the route gains method credibility over time

Deliverables:

- evidence maturity ladder note
- internal review checklist
- pilot response review expectation

Dependencies:

- WP5
- WP6

Acceptance:

- level 1 to level 5 ladder is explicit
- launch claims are bounded by current level
- benchmark or stability claims cannot outrun evidence level
