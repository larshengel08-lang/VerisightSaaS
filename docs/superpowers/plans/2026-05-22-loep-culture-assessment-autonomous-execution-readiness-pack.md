## 5. What Must Be True Before I Can Run Each Block Autonomously

### 5.1 Block A preconditions

- redesign direction is approved
- no request to broaden scope beyond report, deck, and one-pager grammar
- sample-output truth may be changed if it becomes more honest, but not if it creates benchmark or self-serve drift

### 5.2 Block B preconditions

- Block A outputs are either complete or deliberately paused
- PR branch exists and is reviewable
- no unresolved product-scope dispute remains inside the branch

### 5.3 Block C preconditions

- mainline or review branch is stable enough to build on
- the operating model may be clarified and hardened
- legal or procurement direction is not fundamentally disputed

### 5.4 Block D preconditions

- outputs, trust, and operations are strong enough for a real pilot handoff
- any manual/admin-seeded capability is explicitly documented as such

### 5.5 Block E preconditions

- at least one real pilot has happened
- live pilot findings exist in a form that can be translated into product changes

## 6. Human-Only Gates

The following items should not be guessed or silently finalized in autonomous execution.

### 6.1 Legal and commercial truth gates

- final DPA/legal wording if it changes legal position
- final procurement claims if they exceed current documented posture
- any new certification, hosting, or security claim

### 6.2 Live operating ownership gates

- replacing placeholder or role-based owners with real named humans
- real support/escalation ownership for external pilots

### 6.3 Real-world pilot evidence gates

- whether a live pilot is judged commercially successful
- whether live customers accept the output family as premium enough
- whether mature-enterprise positioning may be claimed after pilots

### 6.4 Product-scope expansion gates

- opening benchmark work
- opening self-serve setup or self-serve reporting
- opening broader manager or BU runtime role activation
- opening broader export surfaces than currently governed

---

## 7. Autonomous Stop Conditions

Autonomous execution should stop and surface a decision immediately if any of the following happens.

- a task would require changing the meaning of scores or thresholds
- a task would make optional/governed outputs appear standard
- a task would introduce new result fields rather than redesign existing output
- a task would require a new legal/security claim not already documented
- a task would require activating real customer-facing roles or exports beyond current V1 truth
- a task depends on live pilot evidence that does not yet exist
- a task hits a merge/deploy problem that suggests repo-wide instability outside scoped control
