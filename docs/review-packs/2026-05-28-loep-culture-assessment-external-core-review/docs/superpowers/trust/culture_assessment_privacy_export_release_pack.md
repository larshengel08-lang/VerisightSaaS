# Loep Culture Assessment Privacy, Export, and Release Pack

**Date:** 2026-05-20  
**Status:** trust boundary artifact  
**Primary scope:** `Loep Culture Assessment / Loep Cultuurbeeld`  
**Canonical route id:** `culture_assessment`

---

## 1. Purpose

This pack explains the core trust boundaries for:

- privacy
- exports
- release
- processor and subprocessor posture

for `culture_assessment`.

---

## 2. Privacy and Aggregate Use Position

`Loep Cultuurbeeld` is a group-level culture and engagement baseline.

Core position:

- outputs are designed for aggregate reading
- no respondent-level export is part of normal route operation
- thresholds and suppression are part of methodological quality and privacy discipline

---

## 3. Valid Response Versus Safe Release

These are separate:

- valid response determines whether a response contributes to scoring
- safe release determines whether aggregate results may be shown

Normal output requires:

- valid scoring basis
- closed campaign
- release approval
- threshold-safe aggregate view

---

## 4. Export Boundary Summary

Always forbidden:

- respondent-level export
- unsafe local export below threshold
- manager-limited export of governed segment layers

Governed only:

- segment summary export
- HR appendix export where allowed

Operationally visible:

- export status
- release status
- suppression reason

---

## 5. Release Boundary Summary

Release is governed by:

- closure state
- threshold satisfaction
- suppression rules
- explicit release ownership

Closure alone is not enough.
Release alone is not enough.
Both operational and governance conditions must pass.

---

## 6. Processor, DPA, and Subprocessor Posture

This pack must always be able to answer:

- what controller or processor role is assumed
- whether DPA readiness is available
- who the relevant subprocessor parties are
- what the hosting or data location posture is

V1 rule:

- if an answer is not explicit and supportable, it must not be implied in sales or procurement conversation

---

## 7. Retention Summary

This pack must state:

- what campaign data is retained
- how output artifacts are treated
- how deletion or closure requests are handled
- what must remain for governance or audit context

Retention posture must stay aligned with the trust implementation plan and any legal source of truth.

---

## 8. Access Summary

The pack must explain, in plain language:

- who can see executive output
- when HR may see deeper governed output
- why manager-limited does not get free drilldown
- who approves governed access and release

This pack does not redefine the role model.
It points back to the admin and access control note.

---

## 9. Incident Route

The pack must state:

- where privacy or visibility concerns are raised
- who owns release disputes
- who owns unsafe export escalation

This route must be explicit even if the first pilot remains manually operated.

---

## 10. Approval

Approved when:

- no respondent-level export boundary is explicit
- release and threshold logic are explicit
- controller or processor and DPA posture are explicit
- retention summary is explicit
- access summary is explicit
- incident route is explicit
